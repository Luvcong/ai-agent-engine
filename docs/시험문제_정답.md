## Part 2. 단답 / 코드 교정

### Q11
**Bug 1**: `MemorySaver()`를 요청 및 인스턴스 생성시마다 새로 만들면 멀티턴 관리가 불가능하며, checkpointer는 서버 전체에서 재사용될 수 있도록 분리해야 합니다. 또한 구성한 checkpointer를 agent에 연결하지 않은 것도 수정해야 합니다.

**Bug 2**: 클래스 메서드에 `@tool`을 직접 붙이는 경우 `self`스키마를 전달하려고 하기 때문에 문제가 발생할 수 있습니다.

**Bug 3**: `response_fromat` 오타 -> `response_format=ToolStrategy(ChatResponse)`로 수정해야 합니다.

예시 수정:

```python
from langchain.agents import create_agent
from langchain.agents.structured_output import ToolStrategy
from langchain_core.tools import tool
from langgraph.checkpoint.memory import InMemorySaver

# Bug1 수정
checkpointer = InMemorySaver()


@tool   # Bug2 수정
def search_symptom(symptom: str) -> str:
    """증상으로 질병 정보를 검색합니다."""
    return f"{symptom} 관련 질병 정보"


class MedicalService:
    def create(self, model):
        return create_agent(
            model,
            tools=[search_symptom],
            response_format=ToolStrategy(ChatResponse), # Bug3 수정
            checkpointer=checkpointer,
        )
```

### Q13
문제점:
- 문제 3번, 문제 11번과 동일하게 `process_query()` 호출 때마다 `InMemorySaver()`를 새로 생성하므로 `thread_id`를 전달하더라도 멀티턴 형태 유지가 불가능합니다.
- checkpointer는 서비스 외부 또는 서비스 초기화 시점에 한 번 생성한 후 재사용해야 합니다.

수정 코드:

```python
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

# Bug1 수정
_checkpointer = InMemorySaver()

class AgentService:
    def __init__(self, model, tools):
        self.model = model
        self.tools = tools
        self.agent = create_agent(
            model=self.model,
            tools=self.tools,
            checkpointer=_checkpointer,
        )

    def process_query(self, query: str, thread_id: str) -> str:
        result = self.agent.invoke(
            {"messages": [{"role": "user", "content": query}]},
            {"configurable": {"thread_id": thread_id}},
        )
        return result["messages"][-1].content
```

### Q15
**답변** (3~5문장):

`response_format=ToolStrategy(ChatResponse)`의 경우 LLM의 응답 형태를 개발자가 설정한 포맷대로 생성할 수 있도록 응답 구조를 강제하는 옵션입니다.
`ChatResponse`를 `@tool`로 직접 정의한 방식의 경우 구조화 출력 방식이 아니라 LLM이 선택적으로 호출하는 일반 tool을 추가한 것에 가깝습니다.
`@tool` 방식은 최종 응답 스키마를 강제하는 용도로는 적절하지 않으며, 응답 스키마 검증과 안정화를 위해 `response_format`방식을 권장합니다.
